## 검토 결과

### ✅ 잘 설계된 부분

1. **범위가 명확함** — 발굴 로그 + 메시지 개선만 포함, 성과 통계 대시보드는 Phase 1.5로 분리
2. **기존 로직 비침습적** — 발굴 필터·점수 체계 변경 없이 기록만 추가
3. **실패 안전성 고려** — 로그 저장 실패해도 발굴·전송 동작에 영향 없도록 설계

### ⚠️ 보완 필요 사항

**1. closing_report 연동 방식 구체화 필요**

현재 계획은 "closing_report가 발굴 종목 종가를 채운다"고만 되어 있음.
closing_report는 보유 종목만 API 조회하므로, **발굴 종목이 보유 종목과 다를 경우** 별도 API 호출 필요.
→ 발굴 종목 현재가 조회는 `kis_client.get_current_price(code)` 사용 확인 필요.
→ 장마감 후 호출이므로 당일 종가 = 현재가로 취급 가능.

**2. `disc_price` 수집 방법 명확화**

`intraday_discovery.py`에서 현재 `stck_prpr` 필드를 거래량 API 응답에서 추출 중.
→ 이 값이 실제 발굴 시점 현재가인지 확인 필요 (API 응답의 `stck_prpr`은 현재가 맞음).

**3. 30일 보관 정책 구현 방법**

파일 크기가 무한정 커지지 않도록 정리 로직 필요.
→ 기록 시 30일 이전 항목 필터링 방식이 가장 단순.

**4. 메시지 개선 — 후보 표시 조건 명확화**

"후보 4개 이상일 때" 4~5위 표시라고 했는데:
→ 후보가 정확히 4개면 4위만, 5개 이상이면 4~5위 표시로 구체화 필요.
→ 후보가 3개 이하면 기존과 동일.
