## 잘된 점

- **Approval Workflow 전환이 명시적으로 반영됨** — 제안/승인/체결이 3단계로 분리되고 메시지 근거도 명확

- **MAX_DAILY_LOSS 차단 모드 전환** — sys.exit 대신 상태 플래그로 가는 방향은 서비스 가용성 확보

## 문제점

### 1. 불명확한 부분

**1-1. MAX_DAILY_LOSS 단위가 미정**
초안은 "일일 손실한도 초과 시 차단"만 명시하고 단위가 없음. 절대금액(원)인지 총자산 대비 %인지, 실현손익만인지 평가손익 포함인지 모호함. 시나리오 E에서 -500,000원이라는 예시는 나왔지만 단위 합의가 없음.
→ Stage 4에서 **절대금액(원) + 실현손익만** 기준으로 확정 필요.

**1-2. "1위 종목" 선정 기준**
초안은 `intraday_discovery`의 1위를 그대로 제안한다고 썼지만, 기존 발굴 로직은 "상위 5"만 반환하고 "1위 보장 기준"이 따로 없음. 동점일 경우 처리, 교집합 미충족 시 처리 기준이 플랜에 없음.
→ Stage 4에서 "상위 5 중 1위이되 최소 점수·체결강도·등락률 임계치 미달 시 제안 생략" 조건 명시.

**1-3. 장마감 강제청산 시각**
"15:20 주문(체결 여유 10분)"은 리스크 표에만 등장하고 본문에 없음. 또한 KRX 정규장 마감은 15:30이지만 동시호가(15:20~15:30)가 있어 15:20 이후 주문은 동시호가로 처리됨. 시장가 주문이 동시호가에 어떻게 동작하는지 불명확.
→ Stage 4에서 "15:15 자동 청산 시도(시장가), 체결 미완 시 15:20 이후 동시호가 재시도" 같은 2단 구조로 구체화 권장.

**1-4. `trading_state.json`의 자정 자동 해제 트리거**
"00:00 자동 해제"라고만 적혀 있고 누가(어느 프로세스) 해제하는지 미정. position_monitor 데몬이 자정에 깨서 해제하는지, 별도 launchd 스케줄이 있는지, 아니면 다음 장 시작 시 재초기화인지 모호.
→ Stage 4에서 **position_monitor의 자정 틱에서 해제** 또는 **별도 `reset_daily_counters.py` launchd 00:00 등록** 중 하나로 단순화.

### 2. 기술적 위험

**2-1. 파일 I/O 동시성 미해결**
`position_state.json`, `trading_state.json`, `pending_proposals.json` 3종을 `position_monitor`(10초 루프), `telegram_bot`(사용자 명령 수시), `orchestrator`(체결 직후), `intraday_discovery`(round 종료) 4개 프로세스가 동시에 쓸 수 있음. 파일 락이 없으면 덮어쓰기 손실이 발생할 수 있음.
→ 초안은 "Stage 5에서 확정"으로 미뤘으나, **Stage 4에서 이미 "단일 쓰기 프로세스 원칙"은 결정**해둬야 설계자가 흔들리지 않음. 예: "position_monitor만 쓰기, 다른 모듈은 읽기 전용 또는 명령 큐로 요청".

**2-2. 주문 제출 실패·네트워크 타임아웃 미정의**
KIS API 일시 장애·타임아웃 시 재시도 정책이 전무. 중복 주문(retry → 원 요청도 늦게 성공) 위험.
→ Stage 4에서 "주문 타임아웃 5초, 재시도 없음, 실패 시 텔레그램 알람 + pending 복귀" 같은 기본 정책 명시.

**2-3. 체결 확인 메커니즘**
`order_buy()` 응답만으로 체결 확정으로 간주하는지, 별도 체결 조회 API로 확인하는지 불명확. 시장가는 대부분 즉시 체결이지만 부분체결이 발생할 수 있음.
→ Stage 4에서 "주문 응답 + 잔고 조회로 실제 체결 수량 확인" 정책 명시.

**2-4. 봇 재시작 시 pending 복구 시나리오**
`pending_proposals.json` 영속화까지는 적혀 있으나, 봇이 3분 타임아웃 한가운데에 죽었다가 30분 후 재시작되면 재권유 시각 계산이 무너짐.
→ Stage 4에서 "재시작 시 `pending` 상태이고 `last_sent > 5분 초과`면 즉시 만료 처리 + 다음 round 대기"와 같은 복구 규칙 필요.

**2-5. 쿨다운 5분의 의미 충돌**
안전장치에 `COOLDOWN_SECONDS=300`(매도 후 재매수 쿨다운)과 재권유 타임아웃 3분(180초)이 공존. 매도 후 즉시 같은 종목이 다음 round에서 1위로 또 떠도 COOLDOWN으로 막히는데, 이 경우 제안 자체를 스킵해야 하는지 제안은 하되 승인 시 차단되는지 불명확.
→ Stage 4에서 "COOLDOWN 중인 종목은 제안 단계에서 스킵" 명시.

### 3. 범위 문제

**3-1. P0 항목이 12개로 많음**
초안의 P0는 12개인데, 상호 의존성이 강해 "Phase 2.0"과 "Phase 2.1"로 나눌 여지가 있음. 예:
- **2.0 코어**: 명령 3종, 수량 계산, 가격 가드, 자동 매도 4종, 사후 통지, 계좌 분리, 영속화, 시범운영 (8개)
- **2.1 안전망**: 재권유 상태머신, validate_order 중앙화, MAX_DAILY_LOSS 차단 모드 (4개)

→ Stage 4에서 **내부 릴리스 서브페이즈**로 분할 고려 권장. Phase 2.0 먼저 소액 검증 → Phase 2.1로 안전망 확장. 다만 사용자 요청이 "통합 릴리스"라면 유지.

**3-2. P1의 `/설정` 명령**
P1으로 분류되었으나, 실제로는 **초기 운영 중에도 수치 조정이 필요할 것**. reserve_ratio, MAX_DAILY_LOSS, MAX_TRADES_PER_DAY 같은 값을 코드 배포 없이 바꾸려면 필수.
→ Stage 4에서 P0로 상향 또는 "config 파일 직접 수정 + SIGHUP 리로드"로 단순화된 대안 명시.

### 4. 모호성 / 누락 사항

**4-1. 테스트 전략이 `시범 체결 1주`만 존재**
초안 P0 12번 "초기 운영 검증 절차"가 실제 테스트 전략의 전부. 단위 테스트, 통합 테스트, 시뮬레이션(드라이런) 경로가 없음.
→ Stage 4에서 `trading --dry-run` 모드 추가 제안. 실제 주문 호출 대신 로그만 남기고 가짜 체결가로 position_state 업데이트. 이식 과정에서 가장 중요한 안전망.

**4-2. 로그 정책 / 민감정보 마스킹**
계좌번호·주문번호·체결가 등은 어디에 기록하고 어떻게 마스킹하는지 미정. `logs/trading.log`는 신규인지 기존 `logs/stockbot_error.log`에 묻어가는지 불명확.
→ Stage 4에서 "`logs/trading.log` 신규, 계좌번호는 `******1234` 형태로 마스킹, JSON 구조 로그" 정책 명시.

**4-3. 시간외 거래(NXT) 대응**
현재 stockpilot은 정규장 기준이지만 NXT가 활성화된 시점에는 매수 제안·자동 매도가 어떻게 동작하는가? 초안 리스크 표에만 `is_market_open()` 게이트가 나오고 본문에 없음.
→ Stage 4에서 "Phase 2는 KRX 정규장(09:00~15:30)만 대상, NXT는 Phase 3+" 명시.

**4-4. 봇 다운 시 보유 종목 긴급 대응**
position_monitor가 죽으면 자동 손절이 동작하지 않음. launchd KeepAlive는 초안에 있으나, 재시작 지연(최대 10초) 시 손절 누락 가능성 언급 없음.
→ Stage 4에서 "재시작 직후 현재가 즉시 조회로 '놓친 손절' 판정" 추가.

**4-5. 텔레그램 봇 스팸 방지**
같은 시각에 여러 제안이 몰리면 텔레그램 rate limit(초당 30건)에 걸릴 수 있음. 재권유까지 포함하면 봇 블록 위험.
→ Stage 4에서 "메시지 큐 초당 1건 throttle" 언급만이라도.

### 5. 측정 가능성 / 성공 기준 모호

**5-1. "1주일 무사고 운영"의 정의**
"오발주 0건"은 명확하나 "제어 불능 0건"은 주관적. Stage 4에서 정량화 필요:
- 오발주 0건 (의도하지 않은 종목·수량·방향의 주문)
- `validate_order` 우회 체결 0건
- 봇 재시작 후 pending 복구 오류 0건
- 주문 타임아웃 대비 정상 복구율 100%

## 최종 평가

**Stage 4 진행 가능 여부**: ✅ 가능. 위 16개 수정 제안 중 적어도 **1~12번은 필수 반영**, 13~15번은 권장, 16번은 선택.

**주요 리스크**: 파일 I/O 동시성(2-1)과 테스트 전략(4-1)은 Stage 4에서 결정하지 않으면 Stage 5 설계자가 방향을 잡기 어렵다. 두 축은 Stage 4에서 반드시 결론 내야 한다.
