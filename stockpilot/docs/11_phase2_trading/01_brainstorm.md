# Stage 1: 브레인스토밍 — Phase 2 텔레그램 승인형 매수/자동 매도

> 날짜: 2026-04-22 | 담당: Claude Sonnet (형진님 대화 기반) | Effort: Medium
> 상태: 초안 (형진님 검토/승인 필요)

---

## 1. 문제 정의

stockpilot은 현재까지 **관찰·브리핑 시스템**이다. 장중 종목 발굴(intraday_discovery)까지 자동화되었지만, **매수/매도 실행**은 전부 형진님이 수동으로 HTS/MTS에서 직접 처리해야 한다.

**Phase 2의 목표:**
stockpilot이 발굴한 결과를 기반으로 **실전 소액계좌에서 실주문을 낼 수 있는 텔레그램 명령 체계**를 구축한다. 단, "형진님의 인지부하를 늘리지 않는" 것이 핵심 제약.

---

## 2. 핵심 설계 결정 (Approval Workflow)

브레인스토밍 중 "능동 명령 → 수동 승인" 구조 전환이 가장 중요한 결정이었다.

### 이전 구상 (폐기)
```
형진님 → /매수 005930 10 → 봇 실행
```
문제: 형진님이 "뭘 살지" 먼저 고민해야 함. 인지부하 높음.

### 확정 구조 (Approval Workflow)
```
봇 → "삼성전자 매수?" 제안(근거 첨부) → 형진님 /매수함 or /매수안함 → 봇 실행
```
- 봇이 운전자, 형진님은 승인자(게이트키퍼)
- 매수 근거(체결강도·등락률·점수)가 메시지에 포함 → 충동매매 방지
- 명령 문법 극단적 단순화: `/매수함` `/매수안함` `/종목변경` 3개

## 3. 확정된 결정사항

### 3.1 계좌 구조
| 항목 | 결정 |
|---|---|
| 매매 계좌 | 실전 소액계좌 (별도) |
| Keychain 키 | `KIS_TRADING_ACCOUNT_NO` 신규 추가 |
| 기존 조회 계좌 | 완전 분리 — 리포트·`/잔고`는 기존 유지 |

### 3.2 매수 트리거 & 후보 선정
- 자동 트리거: `intraday_discovery` round 2/4/6/8 종료 직후 **1위 종목**을 봇이 자동 제안
- 대안 선택: 형진님이 `/종목변경` 명령으로 **2~5위 중 선택** 가능
- 매수 제안 메시지에는 발굴 당시 정보(점수·체결강도·등락률·이격도) 포함

### 3.3 수량 자동 계산
```
주문가능금액 = KIS inquire-psbl-order API 조회
유보금     = 총자산 × 10% (현금 유보)
가용금     = 주문가능금액 - 유보금
분할 매수금 = 가용금 × {0.5, 0.3, 0.2}  (5:3:2 분할)
수량      = floor(분할 매수금 / 현재가)
```
형진님은 수량 계산 부담 없이 `/매수함`만 치면 끝.

### 3.4 권유 재발송 상태머신
- 타임아웃: **3분**
- 같은 종목에 대해 **최대 3회 권유**
- **분할 단계별 독립 카운터**:
  ```
  {
    "최초진입":    {count: 0/3, last_sent: ts, status: pending|accepted|declined|exhausted},
    "1차_추가매수": {count: 0/3, ...},
    "2차_추가매수": {count: 0/3, ...}
  }
  ```
- 1차 추가매수 조건(평단 -1~-2% 눌림 + SMA20 위 + 1일 경과) 충족 시 새 카운터 시작
- `position_monitor` 데몬이 30~60초 주기로 pending 권유 점검

### 3.5 가격 이탈 가드
- `/매수함` 수신 시 **현재가 재조회 필수**
- 권유가 대비 **±3% 초과 시 매수 취소** + "권유가 XXX → 현재 YYY (+4.5%), 매수 보류" 알림
- 3분 타임아웃 내라도 급등·급락 국면의 지연 체결 원천 차단

### 3.6 매도 정책 (자동 실행)
- 하드스탑 -3% / 트레일링 / 목표가 +5% / 장마감 강제청산 — **승인 불필요, 즉시 실행**
- 체결 후 사후 통지만 ("삼성전자 -3.1% 손절 완료")
- 이유: 반응성 확보 (자다가 못 보면 손절 타이밍 놓침)

### 3.7 주문 방식
- 시장가 고정 (`ORD_DVSN="01"`)
- 실전 계좌 + `KIS_ALLOW_LIVE_ORDER=1` 가드 필수
- 모의투자 건너뜀 (이미 AT_Project v0.1에서 수개월 테스트 완료)

### 3.8 안전장치 (이식 + 조정)
| 축 | 값 | 비고 |
|---|---|---|
| `COOLDOWN_SECONDS` | 300 (5분 권장) | 매도 후 동일종목 재매수 쿨다운 |
| `daily_pnl` 한도 | `MAX_DAILY_LOSS` 초과 시 주문 기능 차단 | **v0.1의 `sys.exit(0)` 폐기** → 텔레그램 알람 + 다음날 자동 해제 |
| `daily_trade_count` | `MAX_TRADES_PER_DAY` 초과 시 차단 | |
| 동시 보유 종목 수 상한 | 미정 (Stage 5에서 결정) | |
| 현금 유보율 | 총자산 × 10% | 수량 계산에 이미 반영 |

### 3.9 설정 관리
- `data/strategy_config.json`에 새 섹션 `trading` 흡수
- **텔레그램 명령으로 설정 수정 가능** — 예: `/설정 trading.reserve_ratio 0.1`
- 런타임 리로드 (재시작 불필요) — Stage 5 기술설계에서 세부 정의

## 4. 아키텍처 뼈대 (Stage 5 입력용)

```
telegram_bot.py           ← /매수함 /매수안함 /종목변경 수신
    ↕ (Decision 큐)
position_monitor.py       ← [신규 데몬] 손절/익절 감시 + 권유 상태머신
    ↕
orchestrator.py           ← validate_order() 중앙 preflight
    ↓
kis_client.order_buy/sell ← AT_Project v0.1 buy_order 이식
    ↓
position_state.json       ← 분할 가중평단 영속화
```

### 이중 루프 (position_monitor 내부)
- 포지션 0개: 60초 주기 (pending 권유만 점검)
- 포지션 1+개: 10초 주기 (손절/익절 즉시 반응)

## 7. Stage 5에서 확정할 열린 질문

1. **position_monitor 데몬 분리 vs 통합** — 형진님 "잠시 고민 후 결정". v0.2 Decision 패턴으로 보면 분리가 유리 (큐로 결합도 최소화). Stage 5에서 확정.
2. **텔레그램 설정 수정 명령 문법** — `/설정 <key.path> <value>` vs `/설정 <key> <value>` vs 대화형 메뉴. UX와 오타 내성 고려.
3. **동시 보유 종목 수 상한 구체값** — 계좌 시드 규모에 따라 결정. 형진님 초기 시드 미정 상태.
4. **초기 시드 / 종목당 최대 한도 구체값** — 운영 초기에는 보수적으로, 점진 증액.
5. **권유 대기 중인 종목이 여러 개일 때 우선순위** — round 2/4/6/8이 순차로 돌며 각기 다른 종목을 제안할 수 있음. 큐 우선순위 정책 필요.
