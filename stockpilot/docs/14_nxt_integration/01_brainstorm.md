## 2. KIS API 점검 결과 (xlsx 전수 검색)

### 핵심 발견 — `AFHR_FLPR_YN` 파라미터
`주식잔고조회`(TTTC8434R) 요청 헤더의 `AFHR_FLPR_YN` 필드:
| 값 | 의미 |
|---|---|
| `N` | **기본값** — 정규장(KRX) 잔고 |
| `Y` | 시간외단일가 |
| **`X`** | **NXT 정규장 (프리마켓 + 메인 + 애프터마켓 통합)** |

> 문서 원문: "X : NXT 정규장 (프리마켓, 메인, 애프터마켓) ※ NXT 선택 시 NXT 거래종목만 시세 등 정보가 NXT 기준으로 변동됩니다. KRX 종목들은 그대로 유지"

### 보조 API
- `TTTC2210R` — 미체결내역 조회의 KRX+NXT/SOR 통합 모드
- WebSocket NXT API 다수 (실시간 호가/체결/프로그램매매)
- REST 시간외 API: `FHPST02300000`(시간외현재가), `FHPST02320000`(시간외일자별주가) 등

### 결론
- **REST 잔고 조회만으로 NXT 통합 평가손익 확보 가능** (WebSocket 불필요)
- 단순히 기존 호출에 `AFHR_FLPR_YN="X"` 추가하면 충분

## 3. 구현 옵션 (Stage 2에서 비교 예정)

### 옵션 1 — `X` 모드 단일 호출 (가장 단순)
- 잔고 조회를 `AFHR_FLPR_YN="X"`로만 호출
- 모든 시간대(NXT 포함) 통합 잔고
- 장점: 코드 변경 1줄, 정확도 ↑
- 단점: NXT 거래가 없는 날도 X 모드 호출 → KIS 응답 차이 가능성 (검증 필요)

### 옵션 2 — `N`+`X` 듀얼 호출 + 차이 표시
- 두 번 호출하여 정규장(N)과 NXT 통합(X) 비교
- NXT 차이 종목/금액을 별도 섹션으로 표시
- 장점: 사용자가 NXT 거래 여부를 즉시 인지
- 단점: API 호출 2배, 응답 시간 ↑

### 옵션 3 — 시간대 분기 (N 또는 X)
- 정규장 시간(09:00~15:30)에는 N, 그 외(아침/저녁)에는 X
- 장점: 호출 1회 유지 + 시간대별 적정 모드
- 단점: 분기 로직 복잡, KIS 동작과 정확히 일치하는지 검증 필요

### 권고 (Stage 1 시점)
**옵션 2** — N+X 듀얼 호출. 처음에는 사용자에게 NXT 차이가 명시적으로 보이는 게 신뢰 회복에 중요. 추후 NXT 거래가 일상화되면 옵션 1로 전환.
